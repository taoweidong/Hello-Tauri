## 〇、架构约束与优化原则（重要）

本报告的优化方案统一遵循用户明确的架构约束：

- **业务代码尽量使用 TypeScript**；
- **非必要时不引入 Rust**；
- **Rust 仅负责客户端打包（编译产物）与功能透传**——只暴露底层原子能力（如读写字节、解压字节流），不做业务判定。

由此导出的具体规则：文件名净化、路径穿越防护、压缩炸弹大小上限、格式识别、`.tgz` 的 tar 解包等**业务规则一律放在 TS 层**；Rust 命令保持为"薄透传"原语，最多保留结构性安全边界（如 `write_file` 必须落在 `app_data_dir` 沙箱内），**不实现业务逻辑**。下文各优化方案均按此原则给出。

## 二、业务需求符合度（Business Conformance｜对照最新 `docs/业务场景.md`）

> 本维度依据细化后的业务场景文档评估"实现是否满足业务规则"。业务模型的关键点：**压缩包格式固定、解压后文件清单固定**（`VERSION.txt` 必须存在，其余如 `APPLOG1-3`/`MSGLOG1-3`/`sample.json`/`sample.csv`/`sample.txt`/`sample.log`/`sample_table_tree.csv` 为可选但类型固定）；**出现清单外的未知文件时应提示"不支持解压展示"**；每种文件类型对应固定展示形态；且要求 **"一种类型 = 一个解析 TS + 一个展示 Vue 页面"**，并支持后续新类型快速扩展。
>
> ✅ **已满足的业务基础能力**（无需整改，供参考）：
> - `VERSION.txt` 必检 —— `src/core/file-validator.ts:54` `ZipContentValidator` 默认 `requiredFiles=['VERSION.txt']`，缺失即判定压缩包不支持；
> - CSV 表格 + 树形左右联动展示形态 —— `CsvRenderer.vue` + `CsvTreeDetail.vue` + `SplitView.vue` + `csv-row-tree.ts` 已实现左表格右树形的分栏联动，可直接复用于 `sample_table_tree.csv`。

### B1 🟠 P1 — 固定文件清单校验缺失，未知文件未提示"不支持解压展示"

**具体描述**
- 业务要求：解压后文件类型固定，**若出现清单外的新增（未知）文件，应提示"不支持解压展示"**；缺失可选文件属正常场景。
- 现状：`src/core/file-validator.ts:46-96` 的 `ZipContentValidator` **仅校验必需文件是否存在**（默认 `['VERSION.txt']`），**没有维护"允许文件清单"白名单，也不会对多出的未知文件做任何拦截或提示**。
- 未知文件进入解析链路后，被 `src/plugins/registry.ts` 的 `safeParse` **静默回退为 hex 展示**（见 A5，`registry.ts:180-187`），与业务"提示不支持解压展示"的要求**完全相反**——用户看到的是十六进制内容而非"不支持"提示。

**影响**：业务硬性规则未落地；未知/异常文件被伪装成"可展示"（hex），既误导用户也掩盖了压缩包不合规的事实。

**优化方案**
- **优化目标**：在解压/校验阶段对照固定文件清单，对未知文件明确提示"不支持解压展示"。
- **预期效果**：合规压缩包正常展示；含未知文件时给出准确、可读的"不支持"提示，而非 hex 兜底。
- **实施建议**（业务规则放 TS）：
  1. 在 TS 层定义业务清单（如 `src/config/archive-manifest.ts`）：必需文件（`VERSION.txt`）、可选文件模式（`APPLOG*`、`MSGLOG*`、`sample.json/csv/txt/log`、`sample_table_tree.csv`）；
  2. 新增 `ManifestValidator`（实现 `FileValidator` 接口，追加进 `defaultValidators`），对解压条目名逐一比对：命中清单→放行；未命中→返回 `ok:false` 且 message 指明"存在不支持的文件：xxx，不支持解压展示"；
  3. 或在文件树构建后为未知节点标记 `unsupported`，在预览区展示统一的"不支持解压展示"占位组件（替代 hex 兜底）。

### B2 🟠 P1 — `sample.csv` 与 `sample_table_tree.csv` 无法区分，纯 CSV 被误加树形联动

**具体描述**
- 业务要求：`sample.csv` = **纯表格展示**；`sample_table_tree.csv` = **表格 + 树形左右联动**（左表格，每行点击加载右侧树形）。二者是**两种不同的展示类型**。
- 现状检测**仅靠扩展名**：`src/plugins/parser/csv-plugin.ts:9-12` `supportedExtensions:['.csv','.tsv']` + `matchesAnyExtension`，两个文件都命中同一个 `csvPlugin`、走同一个 `CsvRenderer`。
- 而 `src/views/renderers/CsvRenderer.vue:19-41,79-131` 对**所有** CSV 行点击都调用 `extractRowTree` 并进入左表格右树形分栏——即 `sample.csv`（应为纯表格）也被**错误地赋予了树形联动**，`sample_table_tree.csv` 与 `sample.csv` 也**无法按业务区分**。

**影响**：`sample.csv` 展示语义错误（业务要求纯表格）；两种业务类型混为一谈，违反"一类型=一解析TS+一展示Vue"。

**优化方案**
- **优化目标**：将两种 CSV 拆为两个独立业务类型，各自的解析与展示互不干扰。
- **预期效果**：`sample.csv` 纯表格；`sample_table_tree.csv` 表格+树形；符合"一类型一TS一Vue"。
- **实施建议**：
  1. 新增独立类型 `table-tree`：拆出 `src/plugins/parser/table-tree-plugin.ts` + 复用/拆分 `csv-row-tree.ts` 解析 TS + 独立展示页 `TableTreeRenderer.vue`（复用 `SplitView`/`CsvTreeDetail`）；
  2. 因扩展名相同，检测需支持**文件名模式**（`*_table_tree.csv` → `table-tree`；其余 `.csv` → `csv`），见 B4；
  3. 还原 `CsvRenderer.vue` 为**纯表格**（移除行点击树形逻辑），树形联动只保留在 `TableTreeRenderer.vue`。

### B3 🟠 P1 — 无扩展名日志 `APPLOG*/MSGLOG*` 无法识别，回退 hex

**具体描述**
- 业务清单中的 `APPLOG1-3`、`MSGLOG1-3` **无文件扩展名**，业务语义为日志（应按日志拆分表格展示，同 `sample.log`）。
- 现状类型识别**依赖扩展名**：`src/plugins/parser/csv-plugin.ts:11` 等 `matchesAnyExtension(file.name, ...)`、`src/plugins/registry.ts` 的 `getParser(ext)`/`detect`（`endsWith`）。无扩展名文件**无任何插件匹配**。
- 结果：`APPLOG1`/`MSGLOG1` 落到 `safeParse` 的 hex 兜底（`registry.ts:180-187`），用户看到十六进制而非日志表格。

**影响**：业务核心文件类型无法正确展示，日志能力对这些文件形同虚设。

**优化方案**
- **优化目标**：`APPLOG*/MSGLOG*` 正确识别为日志类型并按日志表格展示。
- **预期效果**：无扩展名日志文件复用 `log-parser` + `LogRenderer` 正常展示。
- **实施建议**：为类型识别增加**文件名模式规则**（`APPLOG*`、`MSGLOG*` → `log`），见 B4；或让 `logPlugin.canParse` 支持按前缀匹配（如 `name.startsWith('APPLOG')||startsWith('MSGLOG')`）。必要时对无固定时间戳格式的行，`log-parser.ts:5` 的 `LOG_RE` 需放宽或补充多模式匹配。

### B4 🟡 P2 — 类型识别仅靠扩展名，无法满足"固定文件名→固定类型"映射与快速扩展

**具体描述**
- 业务模型是"**固定文件清单 → 每个文件名/模式对应固定类型**"（`VERSION.txt`→版本、`APPLOG*/MSGLOG*`→日志、`sample_table_tree.csv`→表格树、`sample.csv`→表格…）。
- 现状仅有**扩展名驱动**的检测（`registry.ts` `getParser(ext)`/`detect`、各插件 `matchesAnyExtension`），**无法表达**：无扩展名文件（B3）、同扩展名不同语义（B2 的两种 csv）、按文件名固定映射等业务需求，"快速扩展"也无从谈起。

**影响**：架构的可扩展点与业务扩展方向错位，后续每新增一种"按文件名区分"的类型都要改核心检测逻辑。

**优化方案**
- **优化目标**：建立以"文件名模式 + 扩展名"为输入的统一类型解析机制，支持零核心改动扩展。
- **预期效果**：新增类型只需在清单/插件声明匹配规则，检测链路自动生效。
- **实施建议**：为 `IFileParserPlugin` 增加可选 `matchByName(name):boolean` 或 `namePatterns:RegExp[]`，`registry.detect` 优先按名称模式匹配、回退扩展名；将文件名→类型的业务映射集中到 `src/config/archive-manifest.ts`（与 B1 复用同一份清单）。

---

### B5 🟡 P2 — "一类型=一解析TS+一展示Vue"约束部分未落实

**具体描述**
- 业务硬性约束：每种类型对应**一个解析 TS + 一个展示 Vue**，且便于快速扩展。当前多数类型（json/text/log/csv）已基本遵循（`plugins/parsers/*` + `views/renderers/*`），但：
  - `table-tree` 语义**混入** csv（无独立 parser/renderer，见 B2）；
  - `hex` 是**兜底**而非业务类型，被当作"未知文件"的实际展示（见 B1/A5）；
  - 类型↔解析↔渲染的对应关系分散在各插件，缺少一张可读的"类型注册表"。

**影响**：类型边界不清，新增/维护类型时容易越界耦合，违反业务的清晰扩展约束。

**优化方案**
- **优化目标**：形成"一类型 = 一 parser TS + 一 renderer Vue"的清晰一一映射与集中注册。
- **预期效果**：类型可枚举、可快速扩展；hex 仅用于显式的"不支持/原始字节"场景。
- **实施建议**：整理类型清单（version/text/log/csv/table-tree/json 等），每类补齐独立 parser + renderer，并在 `manifest.ts` 集中登记"类型→插件→渲染器→文件名模式"；hex 从默认兜底改为仅"不支持解压展示"或用户显式选择时使用（配合 B1）。

## 三、安全隐患（Security）

### S1 🔴 P0 — Tauri 自定义 IPC 命令未授权

**具体描述**
- `src-tauri/capabilities/default.json` 的 `permissions` 仅含 `"core:default"`。
- `src-tauri/src/lib.rs:18-29` 通过 `generate_handler!` 注册了 **10 个自定义命令**：`read_file`、`write_file`、`get_temp_dir`、`get_app_data_dir`、`ensure_dir`、`file_exists`、`delete_file`、`mmap_read`、`list_files`、`decompress`。
- Tauri 2 采用 **deny-by-default** 的 ACL 模型，自定义命令必须在 capability 中显式授权，否则前端 `invoke` 调用会在运行时被拒绝（`permission denied`）。
- 当前没有任何 capability 授予上述命令；而 `cache-fs.ts`（`get_app_data_dir`/`file_exists`）、`tauri-adapter.ts`（`read_file`/`write_file`/`decompress`/`mmap_read`/`list_files`）等均是桌面端的真实调用路径。

**影响**：桌面端（缓存初始化、解压、文件读写、列表）在 `tauri dev` / `tauri build` 运行时会**全部失败或报权限错误**，是"打包能跑但功能全废"的最可能根因。

**优化方案**
- **优化目标**：补齐桌面端自定义命令的 ACL 权限，使所有前端调用路径在真实运行时可用。
- **预期效果**：`tauri dev` 下上传→解压→预览→缓存链路不再出现 `permission denied`。
- **实施建议**：
  1. 在 `src-tauri/capabilities/default.json` 的 `permissions` 中追加自定义命令权限（推荐先加 `"hello-tauri:default"`，或逐命令精确授权如 `"hello-tauri:default:read_file"` 等）；
  2. 确认 `src-tauri/Cargo.toml` 中 `tauri` 依赖的 `features` 是否包含生成权限所需的宏（通常默认即可）；
  3. 在 `tauri dev` 中实跑一次完整上传→解压流程验证权限链路；
  4. 后续对写类命令（`write_file`/`delete_file`）按需收窄到最小权限集合。
  5. 授权时保持命令为"薄透传原语"粒度（`read_file`/`write_file`/`decompress` 等），避免新增大量业务特定命令，与"Rust 仅做透传"的定位一致；命名、校验、格式识别等业务规则继续留在 TS。

### S2 🔴 P0 — Gzip 解压路径穿越 + 无炸弹防护

**具体描述**
- `src-tauri/src/decompress.rs:119-140` 的 `decompress_gzip`：`output_name = file_name.unwrap_or("decompressed")`，随后 `output_dir.join(output_name)` 直接 `File::create` 写入。
- 该路径**未做任何净化**：无 `..`/`\`/控制字符拒绝，无 `canonicalize` 前缀校验（对比 `decompress_zip` 在 `:27-90` 的完整防护）；也**没有** `MAX_TOTAL_SIZE` 累计大小上限（zip 在 `:7` 定义了 1GB 上限，gzip 完全没有）。
- `file_name` 来源于用户文件名：`src/plugins/compression/gzip-plugin.ts:14-21` 从用户上传文件名去掉 `.gz/.gzip/.tgz` 后缀得到 `outputName`，传给 `adapter.decompress(data, 'gzip', outputDir, outputName)` → Rust `decompress_gzip`。
- 若用户文件名为 `../../evil.tgz`，`outputName` 为 `../../evil`，`output_dir.join("../../evil")` 会**逃逸 `output_dir`（进而逃逸 `app_data_dir`）** 进行任意位置写入。

**影响**：在 Tauri 端，恶意/畸形 `.gz/.gzip/.tgz` 文件可导致**任意文件写入**（Zip Slip 同类漏洞）或**磁盘被写满**（gzip bomb，无大小上限）。仅 Tauri 端受影响（Web 端写入内存 Map，key 为文件名无穿越），但属高危必须修复项。

**优化方案**
- **优化目标**：使 gzip 解压具备与 zip 同等级别的路径穿越防护与炸弹防护。
- **预期效果**：消除任意文件写入与磁盘耗尽风险，zip/gzip 防护对称。
- **实施建议**（遵循"业务在 TS、Rust 仅透传"原则）：
  1. **把安全校验从 Rust 上移到 TS 业务层**：新增 `src/core/archive-security.ts`，提供 `sanitizeEntryName(name)`（拒绝 `..`、反斜杠、控制字符）与 `enforceTotalSize(total, limit)`（累计大小上限）。zip 与 gzip 共用同一套 TS 校验，不再依赖 Rust 侧各自实现。
  2. **将 `decompress` Rust 命令重构为透明透传**：入参 `(bytes, format)`，返回解压后的条目 `{ name, bytes }[]`（单文件 gzip 返回长度为 1 的列表），**由 Rust 直接返回字节，不自行决定落盘路径、不对文件名做业务判定**。Rust 仅做"解压字节流"这一原子能力。
  3. **TS 侧负责落盘决策**：`use-decompress` / `cache-fs` 拿到解压结果后，先用 `sanitizeEntryName` 处理每个条目名、用 `enforceTotalSize` 累计校验，再调用 Rust 已有的薄原语 `write_file`；`write_file` 的 `validate_path` 只保留"路径必须位于 `app_data_dir` 沙箱内"的**结构性边界**，不实现任何业务规则。
  4. **`.tgz` 的 tar 解包放在 TS**：当前 gzip 插件把 `.tgz` 当单个 gzip 流解压、不解包内部 tar；应在 TS 层用 JS tar 库（如 `tar-stream` / `pako`+tar 解析）完成，Rust 不参与。明确支持范围：要么 TS 实现 tar 解包，要么在上传/检测阶段提示不支持 `.tgz`。

### S3 🟢 P3 — 静态 SVG 经 `innerHTML` 注入（低风险预警）

## 四、性能瓶颈（Performance）

### P1 🟠 P1 — 大文件全量读内存 + IPC 序列化膨胀

**具体描述**
- `src/composables/use-decompress.ts:33`：`new Uint8Array(await archive.file.arrayBuffer())` 将**整包**读入 `ArrayBuffer`，无分块/流式。
- `src/core/cache-manager.ts:89`：缓存归档同样全量 `file.arrayBuffer()` 并经 `Array.from(data)` 序列化写盘。
- `src/core/file-validator.ts:68-73`：`ZipContentValidator` 用 `unzipSync(data)` 将整个 zip **解压到内存**（仅以 200MB 阈值跳过来兜底）。
- `src/adapters/tauri-adapter.ts:31,49` 与 `src/core/cache-fs.ts:78,145`：用 `Array.from(data)` 把 `Uint8Array` 转 `number[]` 经 JSON 走 IPC，对大文件产生约 2–4 倍的序列化开销。

**影响**：大压缩包/大文件内存暴涨、解压/缓存阶段卡顿甚至 OOM；IPC 往返体积成倍放大。

**优化方案**
- **优化目标**：将"全量读内存"改为"分块/流式"，消除 IPC 二进制 JSON 膨胀。
- **预期效果**：大文件不再 OOM/卡顿；IPC 负载与延迟显著下降。
- **实施建议**：
  1. Tauri 2 的 `invoke` 原生支持 `ArrayBuffer`/`Uint8Array` 二进制通道，去掉 `Array.from(data)`，直接传 `Uint8Array`（Rust 侧参数为 `Vec<u8>` 即可）；
  2. 将主数据路径从 `arrayBuffer()` 切换到已存在的 `mmapRead`/`streamRead`（见 A4），按需分块解析；
  3. 对 `unzipSync` 校验改为只读中央目录名（fflate 支持只列文件名而不解压内容），或复用 Rust 端解压结果，避免 JS 侧二次全量解压。

### P2 🟠 P1 — DataTable 虚拟滚动失效

**具体描述**
- `src/components/shared/DataTable.vue:83`：`maxHeight` 默认值为 `'100%'`；`:469-470` 仅在 `useVirtualScroll`（数据 >500 行，`:276`）时启用 `:virtual-scroll` 且 `:max-height="useVirtualScroll ? maxHeight : undefined"`。
- Naive UI `NDataTable` 的 `virtual-scroll` 要求 `max-height` 为**像素数字**才能生效；当值为字符串 `'100%'` 时虚拟滚动**实际不生效**。
- 渲染器 `CsvRenderer.vue:97,112`、`LogRenderer.vue:68` 使用 `<DataTable>` 时**均未传入数值 `maxHeight`**，因此取默认值 `'100%'`。
- 结论：超过 500 行的 CSV/日志表仍会**全量渲染 DOM**；且每次搜索/筛选 `processedData` 都 `props.data.filter(...)`（`:181` 附近）全量遍历。

**影响**：大数据量（数百至数万行）表格卡顿、内存高、滚动不流畅，与架构设计"10 万+ 行虚拟滚动"目标背离。

**优化方案**
- **优化目标**：让大数据表格真正进入虚拟滚动。
- **预期效果**：10 万行滚动流畅，DOM 节点数恒定，内存稳定。
- **实施建议**：
  1. 移除 `maxHeight` 默认 `'100%'`，改为**必填像素数字**（或由父容器 `ResizeObserver` 计算后传入）；
  2. 在 `CsvRenderer`/`LogRenderer` 中基于预览区实际高度传入 `maxHeight`（如 `useResizeObserver` 计算容器高度）；
  3. 大表关闭分页、仅用虚拟滚动；小表保留分页；阈值（500）与分页尺寸（100）提取为 `config/` 常量（见 C3）；
  4. 过滤/搜索结果优先复用 `computed` 避免重复全量遍历，或接入 Web Worker（见 P3）。

### P3 🟡 P2 — 解析在主线程同步执行

**具体描述**：`src/plugins/parsers/csv-parser.ts`、`json-parser.ts`、`text-parser.ts` 等的解析函数为**同步**函数，经 `src/core/parser-engine.ts` 在 `PreviewPane.vue` 的 `watch` 中于**主线程**执行。大文本 `split('\n')` + 逐行解析会显著阻塞 UI 渲染与交互。

**优化方案**
- **优化目标**：将重解析移出主线程，保持 UI 响应。
- **预期效果**：解析期间界面不卡顿，可显示进度。
- **实施建议**：复用设计文档已规划的 `SearchWorker` 思路，将 CSV/JSON/日志解析放入 Web Worker；超大文件采用分块异步解析 + `requestIdleCallback` 增量构建行数组；Worker 与主线程间用 `postMessage` 传递 `Transferable` 的 `ArrayBuffer`/`Uint8Array`。

### P4 🟡 P2 — 大体积解析结果被响应式深度代理

**具体描述**
- `src/components/workspace/PreviewPane.vue:42`：`tab.content = content`，`tab` 属于 `use-tabs.ts` 的 `ref<TabItem[]>`。解析结果（CSV 的 `rows: string[][]`、JSON 整棵对象、hex 整段数据）存入响应式 `ref`，Vue 会对其做**深度响应式代理**。
- `src/composables/use-decompress.ts:71`：`archive.files = tree` 同样把整棵文件树写入 `ref<ArchiveItem[]>`，触达深度代理。

**影响**：海量数组/对象被深度代理带来明显的内存占用与初始化开销，且代理后对象不可被 Worker 结构化克隆（需先降级）。

**优化方案**
- **优化目标**：避免大体积只读数据被响应式代理。
- **预期效果**：内存下降、初始化更快，且便于向 Worker 传递。
- **实施建议**：用 `shallowRef` 包裹解析结果，对的确需要响应式的 UI 状态（如当前行、加载状态）单独维护；对解析得到的原始数据用 `markRaw` 标记；大数组避免放入 `reactive`/`ref` 深层。

### P5 🟡 P2 — TaskScheduler `retry()` 边界缺陷

**具体描述**：`src/core/task-scheduler.ts:74-92` 的 `retry()` 仅对**仍在队列中**的任务有效——若任务已在执行（`running`），`this.queue.find` 返回 `undefined`，原 Promise 不会被 reject，旧任务继续跑，新入队的与旧任务会**重复执行**；且 `retry` 未清理 `promises`/`taskFns` 中 `running` 任务的引用（仅删 `queue`）。

**优化方案**
- **优化目标**：修复重试语义，避免重复执行与状态残留。
- **预期效果**：重试只触发一次、状态一致、无泄漏。
- **实施建议**：维护 `running` 任务映射（id → 控制器），`retry` 时先取消/替换旧任务并清理引用；或引入显式任务状态机（`pending`/`running`/`done`/`failed`）驱动 `retry`/`cancel`。

## 五、架构设计（Architecture）

### A1 🟡 P2 — 缓存层绕过适配器抽象

**具体描述**
- `src/core/cache-fs.ts:9-18,36,59,...` 直接 `await import('@tauri-apps/api/core')` 后自行 `invoke`，绕过 `IPlatformAdapter`。
- 其调用的 `get_app_data_dir`/`ensure_dir`/`file_exists` **不在** `IPlatformAdapter` 接口内（`src/adapters/tauri-adapter.ts` 仅实现 7 个方法：readFile/writeFile/listFiles/getTempDir/decompress/mmapRead/streamRead）。

**优化方案**
- **优化目标**：统一走适配器抽象，使缓存层平台无关。
- **预期效果**：Web/Tauri 缓存行为一致，可单测、可替换后端。
- **实施建议**：将 `getAppDataDir`/`ensureDir`/`fileExists` 纳入 `IPlatformAdapter` 接口并在两个适配器实现；`cache-fs.ts` 改为依赖注入的 `adapter` 而非直接 `invoke`；为 Web 端提供基于内存/`IndexedDB` 的缓存实现。

### A2 🟡 P2 — 双套插件禁用状态不同步，Pinia 冗余

**具体描述**
- `src/stores/app.ts:18,59-80` 维护**独立**的 `disabledPlugins` + `disablePlugin/enablePlugin/isPluginDisabled`。

**优化方案**

### T4 🟠 P1 — 上传校验仅允许 .zip，与 gzip/多格式插件能力矛盾

**具体描述**
- `src/core/file-validator.ts:25-35` 的 `ZipExtensionValidator` 硬拒绝除 `.zip` 外的所有扩展名，提示"当前仅支持 .zip"。
- 但项目已存在 `gzip` 插件（支持 `.gz/.gzip/.tgz`，`src/plugins/compression/gzip-plugin.ts:8`），设计文档更规划了 7z/rar 插件。
- 此外 `.tgz` 经 gzip 插件解压后仅得到**单个解压流**，并不会解包内部 tar（与"压缩包"语义不符），属功能不完整。

**影响**：gzip 等扩展格式虽然在 accept 列表与插件中"看起来支持"，实际在上传校验阶段即被拒绝，**功能不可达**；这是"设计承诺与实现矛盾"的典型技术债务，也误导用户。

**优化方案**
- **优化目标**：上传白名单与已注册插件能力对齐，使扩展格式真正可达且行为正确。
- **预期效果**：`.gz/.gzip/.tgz` 等可被上传并正确解析；未来新增格式零改动即生效。
- **实施建议**：将上传白名单改为**由插件注册表动态生成**（遍历所有 compression/parser 插件的 `supportedExtensions`），移除 `ZipExtensionValidator` 中的硬编码 `.zip`；`.tgz` 的 tar 解包在 **TS 层**用 JS tar 库实现（Rust 仅透传 gzip 解压字节，见 S2），或显式声明不支持 `.tgz` 并在检测阶段给出明确提示。
